## Слайд 2 — О проекте (Цель и функциональность)
**Говорит: Казанцев А. А. (Backend)**

Цель проекта — дать быстрый и удобный поиск по документам университета. Пользователь загружает файлы, система извлекает из них текст, индексирует его и позволяет мгновенно находить нужные фрагменты с подсветкой совпадений.

Ключевые возможности:
- загрузка PDF и DOCX через drag-and-drop;
- автоматическое извлечение текста и индексация;
- полнотекстовый поиск с ранжированием по релевантности;
- история и кеширование запросов;
- запуск всего стека одной командой через Docker Compose.

Работали командой из четырёх человек по ролям: бэкенд, фронтенд, DevOps и QA.

---

## Слайд 3 — Backend: API, индексация, поиск
**Говорит: Казанцев А. А. (Backend)**

Серверная часть написана на Python и FastAPI. Реализован REST API: загрузка документов, поиск и история — всё задокументировано в Swagger.

При загрузке файл проходит валидацию: проверяем формат и размер до 20 МБ и присваиваем документу уникальный UUID. Затем извлекаем текст — pdfplumber для PDF и python-docx для DOCX — и разбиваем его на чанки по 1000 символов с перекрытием 100.

Чанки индексируются в Elasticsearch с русскоязычным анализатором, а поиск идёт запросом multi_match с ранжированием по релевантности. Частые запросы кешируются в Redis на 5 минут; метаданные документов и история хранятся в PostgreSQL.

## Слайд 4 — Frontend: клиентская часть
**Говорит: Канев А. И. (Frontend)**

Клиентская часть — на React и TypeScript. Главный экран — зона загрузки с drag-and-drop и поддержкой множественной загрузки: по каждому файлу виден прогресс и статус — загрузка, индексация, готово или ошибка. Ниже — список загруженных документов с названием, датой и статусом.

Поиск запускается кнопкой «Найти» или клавишей Enter; результаты выводятся карточками — имя файла, номер страницы, найденный фрагмент и оценка релевантности. Совпадения подсвечиваются жёлтым фоном. Есть пагинация по 10 результатов, а вёрстка адаптивная — от 320 до 1920 пикселей.

## Слайд 5 — DevOps: сборка, запуск, мониторинг
**Говорит: Пикуза Д. (DevOps)**

Я отвечал за инфраструктуру. Для бэкенда и фронтенда написаны Dockerfile — фронтенд собирается многоэтапно и раздаётся через Nginx. Весь стек — приложение, база, Elasticsearch, Redis и мониторинг — поднимается одной командой через docker-compose, с healthcheck'ами для правильного порядка запуска.

Настроен CI/CD на GitHub Actions: при каждом пуше автоматически прогоняются линтеры и тесты, а при успехе собираются Docker-образы. Все секреты вынесены в переменные окружения, в репозитории — только шаблон `.env.example`.

Для мониторинга подключены Prometheus и Grafana с метриками по запросам и времени ответа. Скрипт `init.sh` наполняет систему десятью тестовыми документами через API.

## Слайд 6 — Тестирование и качество кода
**Говорит: Магомедов П. А. (QA)**

Я отвечал за качество. Покрытие кода тестами — около 80% при норме 50%: написано 115 юнит-тестов на pytest для парсинга, валидации и сервисов.

Критический путь пользователя — загрузка, индексация, поиск, вывод результатов — проверяется E2E-тестами на Playwright. Нагрузочное тестирование на Locust имитирует 50 одновременных пользователей и собирает отчёт о времени отклика. Качество поиска оцениваем метрикой Precision@3 на эталонных запросах.

Плюс контроль качества кода: PEP 8, линтеры ruff и eslint, docstrings и документация OpenAPI.
